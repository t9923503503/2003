# Инструкция для Claude / Cursor

Краткий operational-гайд для ИИ-ассистентов, работающих в этом репозитории.

---

## 1) Контекст проекта

- Тип проекта: **статическое PWA** (Vanilla JS/HTML/CSS, без сборки).
- Основная логика: `assets/js/**`.
- Состояние хранится в `localStorage`.
- Важные экраны:
  - `assets/js/screens/roster.js`
  - `assets/js/screens/core.js`
  - `assets/js/screens/ipt.js`
  - `assets/js/screens/stats.js`
- Supabase SQL миграции: `supabase_migration.sql`.

---

## 2) Ветки и политика изменений

- `main` — только стабильные изменения (прод).
- `dev` — рабочая интеграционная ветка.
- Фичи делать в ветках от `dev`: `feature/<name>`.

Рекомендуемый flow:
1. `git checkout dev && git pull`
2. `git checkout -b feature/<task>`
3. Изменения + тесты
4. PR: `feature/* -> dev`
5. После проверки: PR `dev -> main`

**Не деплоить в прод напрямую из экспериментальных веток.**

---

## 3) Локальный запуск и проверка

```bash
npm ci
npm run validate:static
npm run test:unit
npm run test:smoke
```

Быстрый локальный сервер:

```bash
npx http-server . -p 8000 -c-1
```

Открыть: `http://127.0.0.1:8000`.

---

## 4) Правила для ИИ при правках кода

1. Сначала понять текущую архитектуру (state/domain/screens/ui).
2. Не ломать старые кнопки/потоки работы, если явно не просили.
3. После правок запускать минимум:
   - `npm run test:unit`
4. Для изменений в static/pwa учитывать кэш Service Worker (`sw.js`).
5. Избегать массового рефактора без необходимости.

---

## 5) Безопасность и секреты

- Не хранить реальные креды в `.md` и коде.
- Файл `SERVER_GUIDE.local.md` — локальный и в `.gitignore`.
- Использовать шаблон: `SERVER_GUIDE.local.md.example`.
- Любые реальные SSH/DB значения держать только локально/в password manager.

---

## 6) Что делать при задаче “задеплой”

1. Убедиться, что изменения уже в `main`.
2. На сервере запустить штатный deploy-скрипт.
3. Проверить health:
   - сайт
   - API
   - системные сервисы
4. Если видна старая версия в браузере — сделать hard refresh (`Ctrl+Shift+R`) и проверить SW-кэш.

---

## 7) Формат отчёта от ИИ

После каждой задачи ИИ должен отдавать:
- какие файлы изменены;
- что именно поменялось по логике;
- какие проверки/тесты выполнены;
- что осталось сделать (если есть).
